## [Architect] Design

**Feature summary:** Phase 4 adds a full-page client-side auth gate to `fno.html` that blocks access until a valid JWT is present in `sessionStorage`, ingests the JWT from the `?token=` query parameter (mirroring the Phase 3 RITA pattern), and adds a "My Portfolio" section to the FnO dashboard that reads the authenticated user's saved allocation portfolio from the existing `GET /api/v1/experience/user-portfolio` endpoint. It also confirms that the localStorage-to-sessionStorage token migration for all FnO JS files is already complete (zero auth-token localStorage references found — `shared/api.js` already covers FnO via re-export, migrated in Phase 3).

**API contract:**

Note: The backend endpoint was delivered in Phase 2. Phase 4 frontend calls it unchanged.

| Field | Value |
|---|---|
| Method | GET |
| Path | `/api/v1/experience/user-portfolio` |
| Query params | none |
| Request body | none |

**Edge cases to handle:**
- No portfolio saved (404): Show `#fno-mp-empty` with "No portfolio saved. Visit the RITA dashboard to create one." Hide holdings table.
- API 401 mid-session: Clear `sessionStorage.removeItem('auth_token')`, re-show `#fno-auth-overlay`. Do not redirect.
- Null or empty holdings array: Guard with `(data.holdings || [])`. Show empty state when array is empty.
- No token on page load (direct navigation): Inline `<script>` in `fno.html` checks sessionStorage synchronously on DOMContentLoaded and shows overlay immediately — no flash of dashboard content.
- Network / 5xx error: `try/catch` shows `#fno-mp-error` with "Unable to load portfolio — " and sets all display elements to `—`.
- `portfolio_id` deferred: Field present in response but not rendered — consistent with Phase 3 advisory.
