## [Architect] Design

### 1. Current State Analysis

`agent-builds.js` calls `apiFetch` twice in `loadAgentBuilds()`:

1. `/agent-ops-data/metrics.json` — maps to `riia-ai-org/agent-ops/metrics.json`, a flat aggregate file served by a FastAPI StaticFiles mount. Top-level keys: `generated_at`, `total_runs`, `per_role`, `per_app`, `grounding_trend`, `failure_modes`, `skill_version_history`, `game_sessions`.

2. `/agent-ops-data/runs/run-{run_id}.json` (N calls, one per run ID) — maps to `riia-ai-org/agent-ops/runs/run-YYYYMMDD-HHMM.json`. Per-run fields: `run_id`, `app`, `request`, `skill_file`, `agents[]`, `overall_status`, `total_tokens_estimated`, `duration_minutes`, `branch`, `merge_status`, `merge_commit`.

Each agent object inside `agents[]`: `role`, `status`, `steps_required`, `steps_completed`, `adherence_score`, `token_estimate`, `grounding_checks` (object of booleans), `failure_modes` (array of strings).

**JS module:** `dashboard/js/ops/agent-builds.js`, exported function `loadAgentBuilds`.
**Section key:** `agent-builds` — registered in `main.js` as `sectionLoaders['agent-builds'] = loadAgentBuilds`.
**Static mount:** `main.py` mounts `riia-ai-org/agent-ops/` as `/agent-ops-data`.
**Current flow:** Browser → GET /agent-ops-data/metrics.json (static) → GET /agent-ops-data/runs/run-{id}.json (static, N calls). No database, no authentication.

### 2. Feature Summary

This migration moves the Agent Build dashboard section from reading raw JSON files served as static assets into the RITA three-tier architecture: two new SQLAlchemy ORM tables store pipeline run data, a new repository class exposes it, and a new Experience-Layer endpoint at `/api/experience/ops/agent-builds` returns the fully aggregated response — replacing both the metrics.json and per-run JSON fetches with a single authenticated API call.

### 4. API Contract

| Field | Value |
|---|---|
| Method | GET |
| Path | `/api/experience/ops/agent-builds` |
| Query params | `limit: int = 20` (ge=1, le=200), `app: str = None` (optional filter) |
| Request body | none |
| Response shape | `AgentBuildsResponse` — see schema below |
| Auth required | No (matches existing Ops Experience pattern) |

**Response schema (`src/rita/schemas/agent_builds.py`):**

```
AgentBuildsResponse
  runs: list[AgentBuildRunOut]   ← sorted by run_id descending
  metrics: AgentBuildMetrics

AgentBuildRunOut
  run_id: str
  app: str
  request: Optional[str]
  overall_status: str
  duration_minutes: Optional[float]
  branch: Optional[str]
  agents: list[AgentOut]

AgentOut
  role: str
  status: str
  token_estimate: Optional[int]
  adherence_score: Optional[float]
  steps_required: Optional[int]
  steps_completed: Optional[int]
  grounding_checks: Optional[dict]
  failure_modes: Optional[list[str]]

AgentBuildMetrics
  total_runs: int
  per_role: dict[str, RoleMetrics]
  grounding_trend: list[GroundingPoint]
  failure_modes: dict[str, FailureEntry]
  skill_version_history: list[SkillVersion]

RoleMetrics
  run_count: int
  avg_adherence_score: Optional[float]
  first_pass_rate: Optional[float]
  avg_token_cost: Optional[float]

GroundingPoint
  run_id: str
  app: str
  grounding_score: float
  checks_passed: int
  checks_total: int

FailureEntry
  total: int
  by_role: dict[str, int]

SkillVersion
  skill_file: str
  last_updated: Optional[str]
  recent_commits: list[str]
```

Metrics are computed from DB rows in-handler (no pre-aggregated JSON). grounding_trend guards against division-by-zero when grounding_checks is empty.

### 8. Edge Cases

1. Empty database — return valid AgentBuildsResponse with runs=[], metrics with total_runs=0 and empty collections
2. Null optional fields (request, branch, merge_commit) — schema marks them Optional[str]; JS guards with `?? '—'`; API must return null not omit the key
3. API call failure — existing try/catch in loadAgentBuilds retained; shows 'Could not load agent-ops metrics' in ab-grid
4. JSON files missing after migration (static mount still active) — grep dashboard/ for agent-ops-data references before removing mount
5. Partial agent data (fewer than 5 agents per run) — renderScorecards already handles missing roles; repo returns as-is
6. Empty grounding_checks — guard: `checks = agent.grounding_checks or {}; checks_passed += sum(1 for v in checks.values() if v is True)`; grounding_score=0.0 not division-by-zero
7. Duplicate run_id on re-seeding — seed script checks existence before insert; idempotent
8. limit=0 or out-of-range — endpoint validates via Query(ge=1, le=200); returns HTTP 422
