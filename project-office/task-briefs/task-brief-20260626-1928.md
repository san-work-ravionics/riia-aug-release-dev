## [Architect] Design

**Feature summary:** Instruments the 7 investment-workflow agents (Financial Goal, Sentiment Analyst, Technical Analyst, Strategy Analyst, Scenario Analyst, Execution Analyst, Outcome Analyst) by recording one row per resolved agent intent from the chat classifier, and surfaces a read-only per-agent KPI dashboard (invocation counts, gap status, outcome-match rate, 30-day trend) in the RITA dashboard so the user can see how each trading-decision agent is performing over time. Delivered in two phases: Phase 1 (data model + fire-and-forget instrumentation) and Phase 2 (read-only endpoint + dashboard).

**API contract (Phase 2):**
| Field | Value |
|---|---|
| Method | GET |
| Path | `/api/v1/experience/rita/agent-performance` (decorator `@router.get("/experience/rita/agent-performance")` on existing rita experience router mounted with `prefix="/api/v1"`; brief's `/api/experience/...` is shorthand — JS MUST call the full `/api/v1/...` path) |
| Query params | none |
| Request body | none |
| Response shape | `AgentPerformanceSummaryResponse` = `{ "agents": [ AgentKpi, ... ] }` — always exactly 7 entries, one per canonical agent. `AgentKpi`: `agent_name: str` (one of 7 canonical names); `gap_status: str` (static label per agent — instrumented / pending-backfill); `invocation_count_30d: int (ge=0)` (rows for this agent in last 30 days; 0 when none); `outcome_match_rate: float\|None` (matches ÷ non-null outcomes; None — not 0.0 — when no backfilled outcomes, so JS renders —); `trend_vs_prior_30d: float\|None` (`(count_30d − count_prior_30d) ÷ count_prior_30d` or signed delta; None when prior window empty) |
| Auth required | no — Experience tier, public read like sibling `/experience/rita/*` routes |
| Read-only | yes — no `db.commit()`; calls `AgentPerformanceRepository(db).summary_for_agents(...)` only |
| Implemented in | `src/rita/api/experience/rita.py` (reuse existing router, no new router file → no new main.py registration; add repo + response schema imports). Not a duplicate — grep for agent-performance / agent_performance returned nothing |

Endpoint always returns all 7 agents by iterating the canonical `agent_name` list and merging aggregation results (missing → `invocation_count_30d=0`, `outcome_match_rate=None`, `trend_vs_prior_30d=None`).

**Data model design (Phase 1):**

*Table `agent_performance`* — ORM model file `src/rita/models/agent_performance.py`, class `AgentPerformance`, inherits `rita.database.Base`. Columns:
| Column | Type | Notes |
|---|---|---|
| `perf_id` | String (UUID) | not null, primary key — string UUID, no auto-increment int |
| `agent_name` | String | not null — one of the 7 canonical agent names |
| `intent` | String | not null — the resolved chat intent that triggered this agent |
| `recommendation` | String | nullable, default NULL — agent recommendation/decision text |
| `outcome_status` | String | nullable, default NULL — realized outcome vs recommendation (match/miss/pending); always NULL in Phases 1+2 (backfillable later) |
| `training_run_id` | String | nullable, default NULL — optional link to `training_runs.run_id`; nullable String link, no enforced FK for SQLite portability |
| `created_at` | DateTime(timezone=True) | not null, `server_default=func.now()` — set by DB, never in Python |

Indexing: composite index on `(agent_name, created_at)` for cheap 30-day-window aggregation.

*Repository* — File `src/rita/repositories/agent_performance.py`, class `AgentPerformanceRepository(SqlRepository[AgentPerformance, AgentPerformanceSchema])`, constructor takes `db: Session` and calls `super().__init__(db, AgentPerformance, AgentPerformanceSchema)`. Extends existing `SqlRepository` base (`rita.repositories.base`). Custom methods:
- `record(agent_name, intent, recommendation=None, outcome_status=None, training_run_id=None)` — best-effort single insert used by the classifier hook; generates `perf_id`, `db.add(row)`; `db.commit()`.
- `summary_for_agents(agent_names: list[str], window_days: int = 30)` — read-only aggregation: per agent, invocation count last 30 days, invocation count prior 30 days (days 31-60), outcome-match counts (matches / non-null outcomes). Must NOT call `db.commit()`.

*Pydantic schema(s)* — File `src/rita/schemas/agent_performance.py`, `model_config = {"from_attributes": True}`. `AgentPerformanceSchema` (row schema, mirrors ORM nullability): `perf_id str`, `agent_name str`, `intent str`, `recommendation str|None=None`, `outcome_status str|None=None`, `training_run_id str|None=None`, `created_at datetime|None=None`. Response/API schemas for Phase 2 (`AgentKpi` + `AgentPerformanceSummaryResponse`) live in the same file.

*Alembic migration* — Revision message `add agent_performance table`. Creates the `agent_performance` table only. No data backfill, no seeding (event table, never seeded). Ordered after current head; applies cleanly on a fresh DB. Register model import in `alembic/env.py` and in `src/rita/main.py` (`# noqa: F401`) so `create_all` includes it on fresh DBs.

**Classifier instrumentation hook (Phase 1 — highest-risk item):**

Location: `src/rita/core/classifier.py`, AFTER intent resolution + mapping to a workflow agent, and AFTER the chat response object is fully constructed — runs on the already-decided result, last thing before return. Never on the critical path of producing the reply. Writes one `agent_performance` row per resolved agent intent: `agent_name` (from mapping), `intent` (resolved key), `recommendation` (decision/signal if present else None), `outcome_status=None`, `training_run_id` (active model build id if available else None).

Fire-and-forget guarantee (all three required):
- Wrap entire hook in broad `try/except Exception` that swallows everything, logs at debug via structlog (never raises, never alters return value).
- Only fires when resolved intent maps to one of the 7 agents (mapping returns None → skip silently).
- Write off the request's critical path — preferred: background thread/executor opening its own session (`SessionLocal()` in worker, `db.close()` in finally) so no DB latency is added and the chat Session is never shared across threads. Fallback: guarded best-effort synchronous insert in same broad catch. Background write is the design choice.
- Hook must not mutate chat response content.

Canonical intent → agent_name mapping (module-level constant `INTENT_TO_AGENT` in `core/classifier.py` or `core/agent_map.py`; engineer aligns left-side keys to classifier's actual resolved-intent identifiers):
| Resolved intent | agent_name |
|---|---|
| financial_goal / goal-planning | Financial Goal |
| sentiment / market-sentiment | Sentiment Analyst |
| technical / technical-analysis | Technical Analyst |
| strategy / strategy-comparison | Strategy Analyst |
| scenario / stress-scenario | Scenario Analyst |
| execution / trade-execution | Execution Analyst |
| outcome / performance-outcome | Outcome Analyst |

The 7 `agent_name` string values are canonical identifiers reused verbatim by the Phase 2 endpoint so the dashboard always shows all 7 agents even with zero rows.

**Edge cases to handle:**
- Empty `agent_performance` table: endpoint returns all 7 agents with `invocation_count_30d=0`; `outcome_match_rate=None` → JS renders `—`, NOT 0%.
- `outcome_status` NULL for every row (no backfill in P1+2): `outcome_match_rate` None (denominator=0 → return None, never divide by zero, never 0.0).
- Prior 30-day window empty (new agent): `trend_vs_prior_30d` None → JS renders `—`.
- API error / network failure in JS: try/catch sets `agent-perf-cards` and `agent-perf-table` to `—`; page does not crash.
- Classifier hook DB failure (write error, schema mismatch, lock contention, session error): broad `except Exception` swallows it; chat response content and latency unchanged; hook logs at debug only.
- Resolved intent maps to no agent (non-investment intent): hook silently skips (mapping returns None) — no spurious rows.
- Unknown/extra `agent_name` in DB not in canonical 7: endpoint iterates canonical list, stray names ignored, dashboard always shows exactly 7.
