## [Architect] Design

**Feature summary:** Phase 3 delivers the "My Portfolio" section to the RITA dashboard — a user-facing allocation builder that lets authenticated users distribute a 100% budget across available instruments, save it as their active portfolio via the backend API, and view their previously saved portfolio on every load. This phase also hardens the JWT token lifecycle by migrating from `localStorage` to `sessionStorage` for the auth token across all RITA JS files, and adds token ingestion from the `?token=` URL parameter directly in `rita.html`'s entry point.

**API contract — Call 1 (load saved portfolio):**
| Field | Value |
|---|---|
| Method | GET |
| Path | `/api/v1/experience/user-portfolio` |
| Query params | none |
| Request body | none |
| Auth required | yes — JWT |
| Expected success | 200 |
| 404 behaviour | no portfolio — render empty builder (not error) |
| Response fields JS reads | `data.holdings` (array), `data.name` (string), `data.portfolio_id` (string), `data.updated_at` (string ISO datetime). From each holding: `h.instrument_id`, `h.allocation_pct` |

| Auth required | yes — JWT |
| Expected success | 201 |
| 422 behaviour | show `err.detail` in `#mp-status-msg` |
| Response fields JS reads | `data.portfolio_id`, `data.updated_at`, `data.holdings` |

4. 422 from POST → show `err.detail` in `#mp-status-msg` with error styling; re-enable save button
