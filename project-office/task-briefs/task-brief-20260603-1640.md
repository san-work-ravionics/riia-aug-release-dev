## [Architect] Design

**Feature summary:** Phase 1 creates the `user_hedge_plans` DB table and two Experience-tier endpoints (`GET` and `PUT` at `/api/v1/experience/fno/hedge-plan`) that allow the server to persist and retrieve a user's hedge plan — recording which instruments are selected for hedging, the coverage level, the active scenario tab, and the fixed 1-year duration. One plan row exists per user (upserted by `key_id`); no frontend wiring is performed in this phase.

**API contract — Endpoint 1: GET /api/v1/experience/fno/hedge-plan**
| Field | Value |
|---|---|
| Method | GET |
| Path | `/api/v1/experience/fno/hedge-plan` |
| Query params | none |
| Request body | none |
| Auth required | yes — JWT |
| Behaviour when no plan exists | HTTP 404, detail: "No hedge plan found" — do NOT auto-create a default row |

Response shape — HedgePlanOut:
| Field | Type |
|---|---|
| key_id | str |
| hedged_ids | list[str] |
| coverage | int (0–100) |
| scenario_tab | str |
| duration | str (always "1y") |
| updated_at | datetime |

**API contract — Endpoint 2: PUT /api/v1/experience/fno/hedge-plan**
| Field | Value |
|---|---|
| Method | PUT |
| Path | `/api/v1/experience/fno/hedge-plan` |
| Request body | HedgePlanCreate |
| Auth required | yes — JWT |
| Upsert behaviour | One db.commit() — insert or update, both paths covered |

Request body — HedgePlanCreate:
| Field | Type / Constraint |
|---|---|
| hedged_ids | list[str] (required) |
| coverage | int ge=0, le=100 (required) |
| scenario_tab | str (required) |
| duration | str \| None = None (accepted but always overwritten with "1y") |

Response shape: HedgePlanOut (same as GET — returns the persisted row confirming saved state)

**Edge cases to handle:**
1. No user_portfolio_keys row for the user — return 404 (not 500), do NOT auto-create a portfolio key
2. hedged_ids contains stale instrument IDs — store as-is; Phase 2 JS reconciles against current portfolio holdings
3. coverage outside 0–100 on PUT — Pydantic ge=0, le=100 raises 422; do not silently clamp
4. duration sent as non-"1y" on PUT — router overwrites with "1y" before persisting; no 422 raised
5. Concurrent PUT from same user — last-write-wins on the PK (key_id); no locking needed
6. Alembic downgrade safety — downgrade() drops only user_hedge_plans; does NOT touch user_portfolio_keys or user_portfolios
